# V5 Memo — Project Handover (2026-07-10)

## 0. Read this first — why the last 9 days stalled

V5 publishing is a **6-stage gauntlet**; a lead must clear **every** stage in one run to publish:

```
1 coverage (5TB sweep, 1525 shards) → 2 mine candidate → 3 publish-quality gate
  → 4 build Researka payload → 5 Researka intake gate → 6 reviewer panel → ACCEPT + DOI
```

Every stage is independently fragile and **different leads die at different stages**. The last ~25
commits are almost all **stage 1 (sweep)** tweaks, while the lead that actually reaches submission
now dies at **stage 5 (payload/intake)** — a stage nobody was looking at. That is the whole reason it
feels like whack-a-mole: fixing the stage that failed last doesn't help, because the next lead fails
elsewhere.

**Do this instead:** pick ONE lead and drive it end-to-end to `accept`, fixing each stage as it dies.
Do not rotate the 12-lead portfolio until one lead publishes. The metformin lead is already the
furthest along (clears coverage + gate, reaches Researka) — start there.

## 2. Blockers, ordered to match the fix plan below

### BLOCKER 1 — Researka intake reject (stage 5) — **blocking the furthest lead today, fix first**
- **Evidence:** metformin `failed_checks: ["submission_template:source_bundle_entry_invalid:3:literal_error"]`,
  `failure_category: "source_bundle_schema"`, `failure_stage: "intake_gate"`.
- **Root cause (found):** source-bundle entry **[3]** is
  `{"doi":"10.6084/m9.figshare.c.3751514_d1","evidence_type":"supplemental","excerpt":"SPIRIT checklist. (DOC 121 kb)"}`
  — a **figshare SPIRIT-checklist supplement**, not real evidence. It is the only entry with
  `evidence_type:"supplemental"` (the other 4 are `"primary"`). Researka's `source_bundle_schema`
  rejects `"supplemental"` as an invalid literal → whole submission bounced before the reviewer.
- **Where:** `src/v5_memo/publisher.py` — `_source_evidence_type` (line ~474) tags figshare DOIs
  `"supplemental"` (line ~482); `build_researka_payload` builds `source_bundle = [_source_bundle_entry(hit) for hit in result.receipts]` and submits it as-is.
- **Fix:** exclude supplemental / non-primary artifacts (figshare datasets, SPIRIT/checklist/DOC supplements)
  from `result.receipts` **before** building the bundle — they are not evidence and they break intake.
  (Secondary: constrain `evidence_type` to Researka's accepted enum so a stray value can't 400 the intake.)
  Guard: if dropping supplementals leaves < 2 direct-human receipts, the candidate should fail the gate
  upstream, not ship a junk entry. **Deeper question:** why did a figshare checklist become a bound
  receipt at all? Filter these out in the retriever/miner too.

### BLOCKER 2 — Claim-card direction mislabel (stage 2/6) — causes reviewer `revise`
- **Evidence:** on 07-05 metformin got reviewer `revise` — receipt `10.2337/db25-1998-lb`
  ("significant reduction in frailty progression", p=0.0222, a **positive** result) was labeled a
  "negative and null signal". Rubric `claim_evidence_alignment=3`.
- **Root cause (proven):** `_polarity`/`_direction_polarity` in `miner.py` score "reduce/reduced" as
  **negative**. The beneficial-reduction carve-out (`_has_beneficial_reduction_context`) misses it because
  (a) it recognizes only verb forms, not the **noun** "reduction", and (b) the guard
  `negative_terms <= _BENEFICIAL_REDUCTION` is all-or-nothing — one incidental negative word
  ("morbidity", "disability") vetoes the beneficial reading.
- **Fix:** add noun/inflected forms (`reduction(s)`, `reduces`, `lowers`, `lowering`, `decrease(s)`,
  `decreasing`) to `_BENEFICIAL_REDUCTION`; relax the subset guard so a beneficial reduction of an
  adverse endpoint in the result sentence wins over stray background negatives.

### BLOCKER 2b — Bundle incoherence (stage 2) — weak/`revise` memos
- Metformin memo pooled `10.1016/j.hlc.2022.06.010` (metformin+HIIT → VO2 in **T2D**) with
  `10.2337/db25-1998-lb` (metformin alone → frailty in **glucose-intolerant**) — different intervention,
  population, endpoint. The memo even prints "should not be pooled unless population, intervention window,
  and endpoint match" then pools them.
- **Fix:** extend the `mixed_outcome_axis_bundle` concept in `gate.py` to reject candidates whose receipts
  don't share intervention + population + endpoint.

### BLOCKER 3 — Sweep engine wedges (stage 1) — flaky coverage, biggest time-sink
- **Symptom:** most leads sit `warming:search_coverage`; health shows `inflight == max_inflight`,
  `key_running=None`, `queued` climbing.
- **Root causes (proven via py-spy earlier):** the "in-flight" slot is a **real sweep stuck in
  `shutil.copy2`** copying a shard from the remote rclone/Storage-Box mount (`_materialized_shard_path`,
  `fullraw_index.py` ~line 1998) — **no timeout**, remote read sampled ~340 kB/s, so a single slow/hung
  shard wedges the sweep and its slot never frees; the dispatcher then refuses all queued sweeps. Plus a
  **cache-key mismatch** — completed sweeps cached under a query key the memo client doesn't re-match
  (the `Match completed V5 sweeps by original query` / `Canonicalize filler terms` commits target this,
  still not solid).
- **Fix:** get a `py-spy dump` when `inflight` is pinned; release `sweep_inflight` in a `finally` on every
  worker exit path; add a timeout to the shard `shutil.copy2`; verify `inflight` returns to 0 and coverage
  advances. **Do not** raise `max_inflight` to mask it (leak is unbounded), and a restart only clears it
  for hours.

### RESOLVED — publish-quality gate (stage 3)
- The `positive_role_direction_mismatch` false-block on positive human RCTs was fixed (`cd26cd2`) and
  verified. Not a current blocker.

## 3. Fix plan (in order — matches blocker numbering above)

1. **Drop supplemental/figshare entries from the source bundle** (`publisher.py`) → unblocks metformin at
   intake. Smallest, highest-leverage: metformin already reaches Researka.
2. **Fix claim-card direction** (`miner.py` beneficial-reduction noun forms + relax subset guard) →
   stops reviewer `revise` on mislabeled positives.
3. **Enforce bundle coherence** (`gate.py`) → stops pooling mismatched studies.
4. **Fix the sweep slot leak + shard-copy timeout** (`fullraw_index.py`) → reliable coverage.
5. **Strategic:** the strict "all 1525 shards, no partial" rule over one ~340 kB/s copy lane is the root
   fragility. Consider pre-warming a small fixed lead set out-of-band, or a high-but-<1525 coverage floor
   with source diversity for the publish path.
